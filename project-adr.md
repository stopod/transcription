# プロジェクト要件 & ADR

---

## 1. プロジェクト概要

- **プロジェクト名**: 議事録要約アプリケーション（仮）
- **リポジトリ名**: `whisper-api`（暫定。Phase 1 完了時に再評価。詳細は ADR-006）

### 1.1 背景・目的

テックイベントに頻繁に参加しているが、議事録・まとめに労力を割いている。
音声を録音し、文字起こし・ノート化までを AI を活用して効率的に行いたい。

### 1.2 スコープ

**含む**:
- 録音 → 文字起こし → 全文保存 までのワークフロー
- Android Chrome での録音・閲覧操作
- 文字起こし結果のコピー／ファイルエクスポート

**含まない（Phase 1 では）**:
- 自動要約（必要があれば Claude.ai に手動投入する運用で代替）
- 他者への配布・共有機能
- iOS Safari 対応（メイン端末が Android のため）

### 1.3 想定ユーザー / ユースケース

- 想定ユーザーは開発者本人のみ。他者利用は想定しないし許容しない。
- ユースケースは、カンファレンスのセッション（30 分〜1 時間）を録音し、終了後に文字起こし全文を確認・保存する。

## 2. 要件

### 2.1 機能要件

| ID | 要件 | 優先度 | 受け入れ基準 |
|----|------|--------|-------------|
| F-01 | スマホブラウザで録音できること | Must | Android Chrome で 1 時間連続録音できる。画面ロック中の挙動を検証し、停止する場合は明示的に注意喚起する |
| F-02 | 文字起こしできること | Must | 日本語 1 時間音声を OSS Whisper で処理し、可読な全文テキストが得られる |
| F-03 | 要約できること | Could | Phase 1 では実装しない。文字起こし全文を Claude.ai に手動投入して代替する（ADR-005） |
| F-04 | 文字起こし全文が端末から取り出せること | Must | 録音から少なくとも 7 日後でも、端末上でテキストを閲覧・コピー・ファイル保存できる |
| F-05 | スマホでメインの操作が完結すること | Should | 録音・文字起こし結果の閲覧・コピーまでをスマホで完結。文字起こし処理自体は PC／自宅サーバー側で動いてよい |

### 2.2 非機能要件

- **プライバシー**: 録音には他参加者の声が含まれうるため、OSS をセルフホストで処理し、第三者クラウドへの音声アップロードは行わない。
- **データ保持**: 録音音声・文字起こしは少なくとも 1 ヶ月、端末またはローカルバックエンドに保持。古いものは手動削除。
- **パフォーマンス**: 1 時間音声の文字起こしは、ジョブ投入から 30 分以内に完了することが望ましい（whisper.cpp medium モデル想定）。
- **オフライン耐性**: 会場で電波が弱くても録音は継続できること。サーバー往復は録音終了後でよい。
- **障害耐性**: 録音中にタブがクラッシュしても、それまでの音声は端末側に残ること。

### 2.3 制約条件

- なるべく低予算で考えている。月額固定費は発生させない。
- 利用スマホは Android。アプリにするとストアに公開する必要があると思っていたが、自分専用ならサイドロードも可能（ADR-001 で PWA を選択）。
- 文字起こしは OSS の **whisper.cpp / faster-whisper** をローカル PC または自宅サーバーで実行。OpenAI / Groq / Hugging Face などの外部 API は採用しない。
- 要約は無料範囲で安定して使える手段が無いため、アプリ要件から外す。必要があれば Claude のサブスクリプションプランで手動要約する。

## 3. アーキテクチャ決定記録 (ADR)

### ADR-001: フロントエンドは PWA とする
- **状態**: 採択
- **背景**: 自分専用で配布の必要がなく、ストア公開のオーバーヘッドを避けたい。アップデートはサーバー側ファイル差し替えだけで済ませたい。
- **決定**: HTML/JS ベースの PWA として実装し、Android Chrome の「ホーム画面に追加」で利用する。
- **影響**: バックグラウンド録音や長時間録音の安定性は OS／ブラウザ依存。事前検証が必要（§4 参照）。

### ADR-002: 文字起こしは faster-whisper を自前実行
- **状態**: 採択
- **背景**: プライバシー（会場の他者音声を含む可能性）と、月額固定費の回避。Python + FastAPI バックエンド（ADR-007）と同言語で扱える `faster-whisper`（CTranslate2 ベース）が、実装統合・GPU 対応・VAD 内蔵の観点で第一候補。
- **決定**: `faster-whisper` を採用し、開発機（RTX 4060 8GB VRAM）で **`large-v3` モデルを既定**として fp16 で動かす。モデルサイズは設定で `medium` などに切り替え可能にする。
- **影響**:
  - 文字起こしは「録音音声をバックエンドに POST → ジョブ実行 → 完了ポーリング → 結果取得」の非同期フロー。録音直後に即時結果は出ない。
  - GPU 利用には `nvidia-cublas-cu12` / `nvidia-cudnn-cu9` の pip パッケージで足りるため、システム CUDA Toolkit のインストールは不要（Driver のみで動作）。
  - 入力音声が WebM/Opus（MediaRecorder の既定）でも、ffmpeg がインストール済みのため faster-whisper 内部で読み込める。

### ADR-007: バックエンドは Python + FastAPI、フロントエンドは Vite + vanilla TypeScript
- **状態**: 採択
- **背景**: バックエンドは faster-whisper との統合のしやすさを最優先。フロントエンドは個人用かつ画面数が少ないため、React のコンポーネント階層は過剰。
- **決定**:
  - バックエンド: Python 3.11 + FastAPI + uvicorn。依存管理は `uv`。
  - フロントエンド: Vite + vanilla TypeScript + `vite-plugin-pwa`。ストレージは `idb`（IndexedDB ラッパ）。
- **影響**: モノレポ構成（`backend/` と `frontend/`）。両者は HTTP API で疎結合。

### ADR-003: 自前バックエンド API を介する構成
- **状態**: 採択
- **背景**: OSS セルフホスト Whisper はサーバーで動かす方が現実的（ブラウザ WASM 版はモデルロードと処理時間に難）。PWA から直接ファイルシステムにも触れない。
- **決定**: 軽量な HTTP API（`whisper-api`）を立て、PWA はこれに対して音声アップロード／ジョブ管理／結果取得を行う。
- **影響**: 自宅サーバー or 個人 PC が起動しているときのみ文字起こしが進む。出先で結果を見るには、その時点でサーバーが起動している必要がある。

### ADR-004: データは端末 IndexedDB ＋ バックエンドのファイルシステムに分散保持
- **状態**: 採択
- **背景**: 録音音声は数百 MB 規模になりうるが、文字起こしテキストは小さい。スマホからすぐ見たいのはテキスト側。
- **決定**:
  - 録音音声 → PWA の IndexedDB に一時保持。アップロード成功後にバックエンドへ移し、端末容量が逼迫したら削除可。
  - 文字起こしテキスト → 端末 IndexedDB と バックエンドの両方に保持。バックエンドを正本とする。
- **影響**: クラウド同期は持たない。別端末からの参照はできない（自分専用なので問題なし）。

### ADR-005: 要約はアプリ要件から外す
- **状態**: 撤回（ADR-008 で覆る）
- **背景**: F-03 の優先度が低く、無料で安定して使える要約手段が現状ない。Claude.ai サブスクで手動要約すれば実用上足りる。
- **決定**: アプリには要約機能を実装しない。代わりに文字起こし全文をコピーしやすい UI を備える。
- **影響**: 将来的に Claude API をバックエンドに足す余地は残す（その時点で新しい ADR を追加）。

### ADR-006: リポジトリ名は当面 `whisper-api` のまま
- **状態**: 暫定
- **背景**: 自前バックエンド API がコアコンポーネントになるため、名前として大きくは外していない。ただし PWA フロントエンドも含むため、別名（例：`meeting-recorder`）の方が表しやすい可能性もある。
- **決定**: Phase 1 完了時に再評価する。それまでは `whisper-api` で進める。

### ADR-008: 要約はローカル Ollama で行う（ADR-005 を覆す）
- **状態**: 採択
- **背景**: 開発機に Ollama が導入され（スタートアップ自動起動、Windows サインイン後は常時稼働）、`qwen2.5:latest`（7.6B / Q4_K_M / 約 4.7 GB）が利用可能になった。費用ゼロ・オフラインで日本語要約が現実的になった。
- **決定**:
  - 文字起こし完了後に **同じワーカースレッドで直列に** 要約を実行する。
  - 既定モデルは `qwen2.5:latest`（環境変数 `WHISPER_OLLAMA_MODEL` で切替可）。
  - 要約フォーマットは「議題ごとの要点 / 決定事項 / アクションアイテム / サマリ段落」の議事録風。
  - **要約失敗はジョブ全体を failed にしない**。文字起こしは保存し、`summary_error` フィールドに失敗内容を記録、status は completed のまま。
- **影響**:
  - VRAM 8 GB に Whisper large-v3 fp16 (5 GB) と Qwen2.5:7b Q4_K_M (5 GB) は同時には乗らない。Ollama の自動 CPU フォールバックに任せて初期は動かす。実運用で遅すぎたら、要約直前に Whisper をアンロードする最適化を打つ。
  - Ollama サービスが停止していると要約失敗（文字起こしは成功）。回復方法は README に記載。

## 4. 検証計画（PoC）

| 項目 | 目的 | 合格条件 |
|------|------|----------|
| Android Chrome での 1 時間連続録音 | PWA の録音継続性を確認 | 画面ロック有・無のそれぞれで結果を記録。ロック中に停止する場合は注意喚起 UI で対応可と判断 |
| whisper.cpp で日本語 1 時間音声を文字起こし | 処理時間と精度の確認 | medium モデルで 30 分以内に完了。固有名詞以外はおおむね正しく書き起こされる |
| IndexedDB に数百 MB の音声を保存 | ストレージ上限と挙動を確認 | 1 GB 程度まで保存・取り出しできる |
| 自宅 NW でのアップロード | 安定性とスループット確認 | 1 時間音声（〜100 MB 想定）を 5 分以内にアップ完了 |

## 5. オープンクエスチョン

- Phase 1 完了後、要約機能を追加する場合の優先度（Claude API 利用可否、コスト許容範囲）
- 自宅サーバーがダウンしている間、録音音声を端末側で保持し続けるポリシー（容量逼迫時の自動削除可否）
- リポジトリ名の最終判断（ADR-006）
